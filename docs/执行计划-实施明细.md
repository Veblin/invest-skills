## Phase 0.5：虚拟环境隔离（v0.2 新增）

> **问题**：Phase 0 原始方案使用 `pip install --break-system-packages`，污染了 Homebrew 系统 Python 环境。
> **方案**：参照 `resources/last30days-skill` 的设计模式，引入 `uv` + `pyproject.toml` + `.venv/`。

**产出物：**
- `pyproject.toml` — 声明依赖（akshare/efinance/yfinance/pandas/requests/pyyaml）
- `.venv/` — uv 自动管理的隔离虚拟环境（`.gitignore` 已排除）
- `.gitignore` — 排除 `.venv/`、`.env`、`__pycache__/`、`evidence/*.json`

**使用方式：**
```bash
uv sync                        # 创建/同步 .venv
uv run python -m scripts.lib.env_check   # 所有 python 命令前加 uv run
uv run pytest                   # 运行测试
```

**核心原则：**
- 所有 `python` 调用统一为 `uv run python` 或 `uv run pytest`
- 系统 Python 保持干净（不安装任何项目依赖）
- CI/新用户拉代码后只需 `uv sync` 即可重现完全相同的环境

### 1.0 Tushare 客户端 + 资金流：`scripts/lib/tushare_client.py` + `scripts/lib/fund_flow.py`

> 基石模块，其他采集模块依赖 TushareClient。编号对齐「执行步骤」Step 1.0。

**tushare_client.py**：

```python
class TushareClient:
    def __init__(self, token: str | None = None)
    def query(self, api_name: str, **kwargs) -> pd.DataFrame
    def is_available(self) -> bool
    def remaining_calls_today(self) -> int
```

Token 有效 → 与 efinance 并行请求（先到先用），不作为前置拦截器；无效或配额耗尽 → 静默降级，报告中标注。

> **修订说明**：原设计将 Tushare 设为优先级 `-1` 前置拦截器。对照 DSA 生产经验，Tushare Token 过期、积分不足、接口迁移频繁，作为唯一前置会造成单点故障。现修正为与 efinance 并列优先级（先到先用），失败不阻断其他数据源。

**fund_flow.py**（与 TushareClient 同属 Step 1.0，因为资金流依赖 Tushare）：


### 1.1b 实时行情：`scripts/lib/realtime_quote.py`

> **修订新增**：原设计缺少实时行情数据源规划。对照 DSA 生产架构，实时行情需要独立于历史 K 线的 fallback 链。

```python
def get_realtime_quote(code: str) -> dict
    # 返回: {price, change_pct, volume_ratio, turnover_rate,
    #        pe_ratio, pb_ratio, total_mv, circ_mv, amplitude, source, fetched_at}
```

**Fallback**：

```
腾讯财经(tencent) → akshare(新浪) → efinance → akshare(东财) → 使用最近K线收盘价
```

- 腾讯财经单股查询最稳定（DSA 推荐首选），无需全市场拉取
- efinance/akshare_em 会拉全市场数据，仅在批量分析时做预取优化（参考 DSA 的 `prefetch_realtime_quotes` 策略）
- 全部不可用时使用最近 K 线收盘价，标注 `[非实时，基于 {date} 收盘价]`
- 盘中实时技术面（MA/多头排列）用实时价计算；收盘后自动切换到收盘价

### 1.10 主编排 + 测试入口 + 安全消毒：`scripts/data_pipeline.py`

> 依赖所有 1.0–1.9 模块就绪，是 Phase 1 的最后一步。

```python
def build_collection_plan(symbol: str, asset_type: str, dims: list[str] | None) -> dict

```

**行为**：

- 按维度并行调用 `lib/` 模块（`concurrent.futures`）；单维失败不阻断其他维
- 统一附加 `_meta`：`source`、`source_group`、`fetched_at`、`confidence`、`fallback_chain`、`latency_ms`、`success`、`error_type`（完整契约见 [架构关系 §2.2](./执行计划-架构关系.md#22-采集模块内部契约)）
- **`_meta` 安全消毒**（P0-04）：在 `collect_dimension` 返回前调用 `sanitize_meta()`，对所有 `_meta` 字段执行：
  - 敏感 Key 名检测：字段名含 `token`/`secret`/`key`/`password`/`authorization` → 值替换为 `[REDACTED]`
  - Bearer Token 正则清洗：`Bearer xxx` → `Bearer [REDACTED]`
  - URL 脱敏：异常信息和原始响应中的 `https://...` URL → `[REDACTED_URL]`
  - 长度截断：`error_detail` 超过 300 字符自动截断
- **采集遥测**（Day 1 写入）：每次 fallback 记录 `{source, latency_ms, success, error_type}`，即使失败也写。落地后校准 S/A/B 分级的唯一可靠依据
- 集成 `TushareClient.is_available()` 决定数据源优先级（并列而非拦截）
- 采集顺序默认遵循框架优先级：**财务 → 基本信息 → 行业 → 估值/技术 → 机构 → 情绪 → 宏观**（`--dim` 可裁剪）

- Step 0 预研清单：代码映射 → 行业 → 相关 ETF → 覆盖机构 → 输出采集计划摘要

### 3.2 决策树

```
/invest-A {topic} [flags]
    ↓
Step 0 预研
    解析代码与市场 → 识别 asset_type（stock/hk/etf）
    检查 TUSHARE_TOKEN、FRED_API_KEY
    解析 --dim / --with-macro / --deep
    输出采集计划摘要（用户可确认）
    ↓
Step 1 采集
    data_pipeline.collect_all(...) 或 MCP 等价调用
    保存 evidence/raw.json（若可写文件）
    ↓
Step 2 分析
    按维度加载 strategies/*.yaml
    严格 instructions；无数据则「数据不可得」
    ↓
Step 3 报告
    report_render.render_report → Markdown（+ 可选 HTML）
```

### 3.3 错误处理契约

| 场景 | 行为 |
|------|------|
| akshare 失败 | efinance → baostock → 标注不可得 |
| Tushare 未配置 | 免费路径，报告注明 |
| FRED 无 Key | yfinance → akshare |
| 单维全失败 | 标题加 ⚠️，列出 `attempted_sources` |
| 网络超时 | 重试 3 次指数退避 |
| **异常信息含 Token** | `_meta.error_detail`/`_meta.raw_response` 经 `sanitize_meta()` 消毒后再写入（P0-04 安全红线） |
| ETF 代码（MVP） | 运行 basic_info + etf_guide 导读，注明 Phase 6 |
| Hermes 无 Python | 走 FinanceMCP / WebSearch，功能矩阵见主文档 |
