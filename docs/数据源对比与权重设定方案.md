### 1.2 DSA 数据源优先级经验

DSA 经过大量生产验证后，确定的优先级顺序（数字越小越优先）：

**日线数据（A股）**：
```
0. EfinanceFetcher      ← 东财，最稳定（DSA实测首选）
1. AkshareFetcher       ← 东财+新浪+腾讯多源，备用
2. PytdxFetcher          ← 通达信，可自建服务器
3. BaostockFetcher       ← 最稳定但数据延迟1天
4. YfinanceFetcher       ← 全球覆盖但A股质量差
[有Token时插入] TushareFetcher(P0) ← 需要2000+积分才稳定
```

**实时行情（A股）**：
```
tencent > akshare_sina > efinance > akshare_em > tushare
```
腾讯财经单股查询最稳定，efinance/akshare_em拉全量易被封。

**关键教训**：
1. efinance 在 A 股日线上比 akshare 更稳定（EP 设计相反）
2. Tushare 不应作为唯一前置拦截器——Token 过期、积分不足、接口迁移频繁
3. 需要"实时行情"和"历史K线"两条独立 fallback 链
4. 中文新闻搜索必须有多源：Bocha（中文优化）不可替代

## 二、EP 数据源可用性评估

### 2.1 数据源可靠性分级

基于 DSA 生产经验和公开可用信息，对 EP 数据源按可靠性分级：

| 分级 | 定义 | 数据源 |
|------|------|--------|
| **S 级** | 免费、无需Token、长期稳定、商业级可靠性 | efinance（东财）、baostock |
| **A 级** | 免费无需Token，偶有维护窗口 | akshare、yfinance |
| **B 级** | 免费但需Token，Token获取容易 | Tushare（注册即得，但高积分才稳定）、FRED |
| **C 级** | 免费但需Token+信用卡/特殊条件 | Tavily（免费层500次/月）、Finnhub（60次/分钟）|
| **D 级** | 需付费Token或接口不稳定 | 部分Tushare高级接口、TickFlow |
| **X 级** | ToS风险或法律灰色地带 | 雪球爬虫（EP已排除 ✅）、部分未授权API |

### 2.3 关键差异：EP 需修正的设计

| # | 问题 | EP 当前设计 | DSA 实践 | 建议 |

## 三、信息源权重初始设定

### 3.1 权重设计原则

1. **可靠性优先**：S级数据源的数据在报告中给予最高权重引用
2. **降级透明**：使用低可靠性数据源时，必须在报告中降低该维度的置信度标注
3. **交叉验证加分**：两个以上独立数据源交叉验证一致的数据，置信度提升一档
4. **时效性惩罚**：数据超过一定时间窗口后自动降权
5. **无数据不造假**：全部不可得时明确标注"数据暂不可得"，不编造

### 3.2 数据源可信度评分（Source Credibility Score）

| 数据源 | 可信度 | 理由 |
|--------|--------|------|
| efinance（东方财富） | ★★★★☆ 4/5 | 国内最大财经数据平台，免费无需Token，DSA验证最稳定，但非官方一级数据源 |
| akshare | ★★★☆☆ 3/5 | 开源社区维护，多数据源聚合，偶有接口变更导致的breaking change |
| baostock | ★★★★☆ 4/5 | 学术用途，数据质量高，但更新滞后（T+1） |
| Tushare Pro | ★★★★☆ 4/5 | 需Token+积分，高级接口质量好，但免费层限制多 |
| yfinance（雅虎财经） | ★★★☆☆ 3/5 | 全球覆盖，A股数据质量一般，港股尚可 |
| FRED（美联储） | ★★★★★ 5/5 | 官方一级数据源，权威性无可争议 |
| Tavily Search | ★★★☆☆ 3/5 | AI优化的搜索API，结果质量高但有配额限制 |
| WebSearch（Claude内置） | ★★★☆☆ 3/5 | 无需额外配置，但结果质量依赖搜索引擎 |
| Bocha（中文搜索） | ★★★☆☆ 3/5 | 中文搜索优化，国内信息覆盖好 |
| 东方财富股吧 | ★★☆☆☆ 2/5 | 散户情绪指标，噪声大，但作为情绪数据有参考价值 |
| 腾讯财经（实时行情） | ★★★★☆ 4/5 | 大厂API，稳定性好，但无官方文档保障 |
| 新浪财经（实时行情） | ★★★☆☆ 3/5 | 老牌接口，稳定性尚可 |

### 3.3 按分析维度的信息源权重分配

报告中的每个维度，按照"可靠数据源占比"确定该维度的整体置信度：

| 维度 | 数据源组合 | 维度权重（学习分析用） | 最低可用标准 |
|------|-----------|---------------------|-------------|
| **一、基本信息** | efinance + akshare + baostock | ★★★★ 4/5 | 至少1个S/A级数据源成功 |
| **二、财务报告** | efinance + akshare + baostock → yfinance | ★★★★★ 5/5 | 至少1个数据源返回最近2年数据 |
| **三、行业产业链** | akshare(申万) + WebSearch | ★★★ 3/5 | 至少获得行业分类 |
| **四、估值** | akshare + efinance + 知识库分位参考 | ★★★★ 4/5 | 至少有PE/PB当前值 |
| **四(b)、市场状态** | efinance(K线) + 实时行情(tencent) | ★★★ 3/5 | 至少获得MA数据 |
| **五、机构分析师** | akshare(调研/北向) + WebSearch(研报) | ★★★ 3/5 | 至少1类数据可得 |
| **六、情绪舆情** | akshare(股吧) + Tavily/WebSearch(新闻) | ★★ 2/5 | 注意：此维度本身噪声大 |
| **七、宏观政策** | FRED + yfinance + akshare + WebFetch | ★★★★ 4/5 | FRED不可得时降为★★★ |

> **说明**：以上权重不表示"股票好坏"，仅表示"当前分析所用数据的可靠程度"。权重附在报告各维度标题后，如 `## 二、财务报告 ★★★★☆`。

### 3.4 交叉验证权重加成规则

当同一指标从两个以上独立数据源获取且数值偏差 <5% 时：

| 交叉验证数据源数 | 置信度调整 |
|-----------------|-----------|
| 2个独立源一致 | 该指标置信度 +0.5★ |
| 3个及以上独立源一致 | 该指标置信度 +1★（上限 5★） |
| 2个源偏差 >10% | 标注 ⚠️ "数据存在分歧"，取中值，置信度 -1★ |
| 仅1个源可用 | 保持原始评级，标注 `[单一来源]` |

### 3.5 时效性降权规则

| 数据类型 | 新鲜窗口 | 过期降权 |
|---------|---------|---------|
| 实时行情（价格/量比） | 盘中实时 | 收盘后标注 `[收盘价]`，次日开盘后标注 `[数据延迟]` |
| 日线K线 | T+0（当日收盘后） | T+2 标注日期，不降权 |
| 财务报表 | 最新季报后90天 | 超过90天标注"最新财报为YYYYQQ"，降权 -0.5★ |
| 北向/南向资金流 | T+0 | T+1 标注日期，T+3 标注"三日延迟" |
| 机构调研记录 | 30天 | 30-90天标注日期，>90天 降权 -1★ |
| 新闻/舆情 | 7天 | 7-30天标注日期，>30天 降权 -1★ |
| 宏观数据(CPI/PMI) | 发布后30天 | 标注"最新数据：YYYY-MM"，不降权（宏观数据更新频率固定） |
| 分析师一致预期 | 90天 | 90-180天标注，>180天 降权 -1★ |

## 四、数据源不可用时的系统性降级策略

### 4.1 降级分层模型

```
Layer 0: 最佳路径（S/A级数据源全部可用）
    ├─ 报告完整度：100%
    ├─ 各维度置信度：★★★～★★★★★
    └─ 采集耗时：快（最优路径）

Layer 1: 部分降级（S级可用，A级部分不可用）
    ├─ 报告完整度：85-95%
    ├─ 受影响维度标注"部分数据不可得"
    └─ 采集耗时：中（有重试）

Layer 2: 中度降级（仅 A/B 级可用）
    ├─ 报告完整度：60-85%
    ├─ 整个不可得维度标注 ⚠️ 并跳过
    ├─ 尾部汇总：列出所有 attempted_sources
    └─ 采集耗时：较慢（多重试）

Layer 3: 重度降级（仅知识库+WebSearch可用）
    ├─ 报告完整度：30-60%
    ├─ 仅能提供：基本信息 + 知识库框架 + WebSearch 公开信息
    ├─ 报告首段明确声明："⚠️ 以下分析基于有限数据源"
    └─ 采集耗时：快（跳过所有失败源）

Layer 4: 完全离线（无网络/所有源不可用）
    ├─ 报告完整度：<30%
    ├─ 仅输出知识库内容 + 框架教学
    └─ 建议用户检查网络或稍后重试
```

### 4.2 用户可见的降级提示语

| 场景 | 提示语模板 |
|------|-----------|
| 单维度不可得 | `⚠️ 「{维度名}」数据暂不可得（尝试了 {sources}），跳过此维度分析。` |
| 数据源降级 | `[数据来源: {source}（降级路径: {fallback_chain}）]` |
| 时效性过期 | `[数据日期: {date}，已超过新鲜窗口，仅供参考]` |
| 交叉验证失败 | `⚠️ {指标名} 在不同数据源间存在 >10% 偏差（{源1}: {值1}, {源2}: {值2}），取中值，请独立验证。` |
| 全部不可得 | `🚫 当前所有数据源不可用。以下内容基于知识库框架，不包含实时数据。建议检查网络或稍后重试。` |

### 4.3 `_meta` 字段毒性过滤（P0-04 直接相关）

参照 DSA 的 `sanitize.py` 实现，EP的 `_meta` 字段在渲染前必须经过：

1. **敏感Key过滤**：任何包含 `token`/`secret`/`key`/`password`/`authorization` 的字段名，值替换为 `[REDACTED]`
2. **URL脱敏**：`_meta.error_detail` 中的 URL 替换为 `[REDACTED_URL]`
3. **Bearer Token脱敏**：`_meta.raw_response` 或异常堆栈中的 `Bearer xxx` 替换
4. **长度截断**：`_meta.error_detail` 超过 300 字符自动截断

| 配置文件 | 内容 |

| `config/source_credibility.yaml` | 数据源可信度 ★ 评分、`source_group` 分组、`internal_score`（0–100）、tier 分级（S/A/B/C/D） |
| `config/dimension_baselines.yaml` | 七维度 + ETF 变体的 `baseline_stars`、`internal_weight`、`min_viable_sources`、分阶段启用规则 |
| `config/cross_validation_rules.yaml` | 交叉验证阈值（全局 5%/10% + 按 metric 类型细化）、时效性降权窗口、质量等级映射 |

**读取方式**：`quality_scorer.py` 在初始化时 `yaml.safe_load()` 上述文件，不硬编码任何数字。

### 6.2 source_group 定义

为实现交叉验证时的"独立源"判定，每个数据源归属到一个 `source_group`：

| source_group | 成员 | 特征 |
|-------------|------|------|
| `eastmoney` | efinance, akshare, akshare_em | 东方财富系，接口同源 |
| `sina` | tencent_finance, akshare_sina | 新浪/腾讯系 |
| `official` | fred, tushare | 官方/一级数据源 |
| `academic` | baostock | 学术数据源 |
| `global` | yfinance, finnhub, alphavantage | 国际通用源 |
| `search_ai` | tavily, bocha, websearch_claude | AI搜索/搜索引擎 |
| `sentiment` | eastmoney_guba, pytrends | 散户情绪 |
| `commercial` | longbridge, tickflow, pytdx | 商业/付费接口 |

**规则**：同一 `source_group` 内的两个源不算独立源（如 efinance + akshare），不触发交叉验证加分。异组双源一致才加分。

### 6.3 采集遥测设计

从 Day 1 开始，每次 fallback 写入以下遥测字段（存在 `_meta` 中，落地到 `evidence/raw.json`）：

```python
{
    "source": "efinance",
    "source_group": "eastmoney",
    "latency_ms": 320,
    "success": true,
    "error_type": null,
    "fetched_at": "2026-06-10T14:30:00",
    "fallback_chain_position": 1,  # 第几个尝试成功（0-based）
}
```

**用途**（后续校准）：

| 指标 | 用途 |
|------|------|
| 各 source 成功率 | 校准 S/A/B 分级 |
| P95 延迟 | 判断是否需要调整 Fallback 顺序 |
| Fallback 触发率 | 链是否过长、是否需要加 Pytdx |
| 同源双源一致率 vs 异源一致率 | 决定是否启用 source_group 加分 |
| Layer 分布（0–4 各占多少） | 校验降级模型完整度百分比是否贴近现实 |

### 6.4 分阶段启用路线图

```
MVP (Day 1)
  ├─ _meta 四字段（source, fetched_at, confidence, fallback_chain）
  ├─ 单一来源标注 [单一来源]
  ├─ Fallback 链记录 + Layer 降级提示语
  └─ 采集遥测写入（latency_ms, success, error_type）

v0.2（遥测积累 ≥500 次采集后）
  ├─ 维度 ★ 展示（基于 dimension_baselines.yaml 初始值）
  └─ 时效性标注（可先只标注不降权）

v0.3（遥测积累 ≥2000 次采集后）
  ├─ 交叉验证加分
  └─ 分歧告警（≥10% 偏差时）

v1.0（生产环境充分验证后）
  ├─ 与 DSA 式 0–100 内部分打通
  └─ prompt 护栏驱动（confidence_level 不得为「高」等硬约束）
```

**原则**：架构一次到位，规则逐步收紧。配置外置保证校准不需要改代码。

**职责**：

def calculate_overall_quality(
